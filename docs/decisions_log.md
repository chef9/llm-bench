## 22. Key Architectural Decisions Log

This section records the key decisions made and the reasoning behind them, in the order they were made.

**u128 bitfields for binary net engine**
Chosen for Apple M4 Pro performance — native 128-bit operations. Supports 128 places and 128 transitions per net, which is sufficient for any individual subnet in a hierarchical model. Decision: keep.

**No fixed ontology**
Semantic grounding toward human users is handled by Claude, not by OWL/RDF or a proprietary schema. NeMo's internal u64 IDs are unambiguous by structure. A fixed ontology would constrain modeling flexibility and couple the internal representation to a specific semantic framework. Decision: Claude handles semantics above a stable append-only ID registry.

**Synthesis in its own crate**
`shared/synthesis` depends on `shared/core`, not vice versa. This keeps the core engine pure and allows simulation and validation to consume core without synthesis overhead. Decision: maintain strict crate dependency direction.

**Crate naming: avoid shadowing `core`**
Naming a crate `core` shadows Rust's built-in `core` crate, requiring `std::fmt` prefixes throughout. The crate is named `nemo-core`. Decision: always use project-namespaced crate names.

**Unikernel via Nanos/OPS, not Hermit**
Nanos runs any static ELF binary without a special Rust compilation target. Hermit requires `x86_64-unknown-hermit` and has limited ecosystem compatibility. For NeMo's use case, Nanos provides the same timing guarantees with broader portability. Decision: Nanos for production, same binary runs on Linux for development.

**Binary nets as control layer, not primitive**
The initial instinct was to use binary nets as building blocks and compose richer behaviour above them. The correct architecture is the inverse: binary nets govern which net variant handles which subtask. Wherever binary nets can express the behaviour, they should — they are faster and better understood. Richer variants are invoked only for tasks that require them. Decision: binary nets are the control layer at the top of the hierarchy.

**Knowledge graph replaces separate database**
Instance data lives directly in the knowledge graph structure, identified by global u64 IDs. There is no separate relational or document database layer. The global ID space is the schema. This eliminates a dependency tier and keeps the data model unified. Decision: knowledge graph as the sole persistence layer.

**Separate mapping layer between nets and knowledge graph**
Tokens in colored Petri nets carry global IDs pointing to knowledge graph nodes, not embedded data. A separate mapping layer mediates between net state and graph nodes, following the same pattern as proclet ID mapping. Decision: nets stay clean; data lives in the graph.

**Flutter as dumb view layer**
All domain logic lives in Rust. Flutter renders engine output and forwards user events. The app is domain-agnostic — it does not need to be updated when a new proclet is deployed or NeMo is applied to a new domain. Decision: no business logic in Flutter; all logic in `shared/core`.

**Rinf: Serde not Protobuf**
Rinf v8+ uses Serde. No `.proto` files, no `protoc` installation, no code generation step beyond `dart run rinf gen`. Signal structs are native Rust with derive macros. Decision: Serde throughout, consistent with the rest of the Rust codebase.

**Quantum-compatible architecture by design**
NeMo's binary Petri net foundation (u128 bitfields), bipartite graph model, and synthesis-as-optimisation framing all have direct structural alignment with quantum computing algorithms — Grover's search for reachability, QAOA/annealing for synthesis QUBO, and quantum walks for knowledge graph traversal. The architecture preserves quantum compatibility without creating quantum dependency: explicit bipartite structure, backend-agnostic simulation interface, QUBO-expressible synthesis constraints, and probability-weighted reachability envelopes. Quantum backends will be substituted via the same compute-backend abstraction used for GPU/NPU. Decision: design for quantum compatibility now; integrate quantum backends when hardware matures to the required logical qubit threshold.

**Direct code generation via low-level Petri nets**
NeMo's end-to-end verification guarantee is only complete if the execution layer is itself generated by the verified model. Any external compiler introduces an unverified translation gap. Code-generating nets — a specialised net variant whose transitions emit ARM64 instruction bytes into executable unikernel RAM regions — close this gap. The synthesis rules for code-generating nets encode register allocation, memory safety, and control flow integrity as structural constraints, preserving the correctness-by-construction guarantee from process model to executing binary. ARM64 is the primary target for its fixed 32-bit instruction width and orthogonal register file. The bootstrapping base is a small, formally-verifiable Rust implementation of the net engine. Decision: implement code-generating nets as a distinct variant in `shared/synthesis`; ARM64 first; all generated code traceable to a synthesis derivation in the append-only registry.

**Cryptographic access control — no trusted platform operator**
Policy-based access control requires trusting the operator. Cryptographic access control with customer-held keys eliminates this trust requirement. Data is encrypted with keys derived from an organisation-held OMK before entering the storage layer. NeMo's operations team cannot read customer data. Personal data is isolated to a Personal Key branch not derivable from any Domain Key. The key hierarchy (HKDF-SHA256, OMK → Domain Keys → Proclet Keys, separate Personal Key branch) mirrors organisational access structure. Storage layer sees encrypted blobs; graph topology visible, graph semantics are not. All writes are Ed25519-signed; the append-only log is tamper-evident by construction. Cryptographic primitives live exclusively in `shared/crypto`; no plaintext path exists through the storage API. Session keys are ephemeral and unikernel-bound. Decision: `shared/crypto` as sole crypto location; AES-256-GCM for node encryption; Ed25519 for log signing; HKDF-SHA256 for key hierarchy; X25519 + ChaCha20-Poly1305 for key distribution; no plaintext storage path; no exceptions.

**Generative subnet construction, not expert curation**
Seeding the subnet library through domain expert interviews produces subnets shaped by practitioner assumptions and institutional habit — the equivalent of asking a professional cook for a recipe. The correct approach is to generate subnets from atomic nets using the same synthesis rules that govern process model composition, validate them using the simulation layer, and use synthesis success rates as the pruning signal. Claude labels generated subnets by working from formal structure upward to natural language, not from expert description downward. Domain partners validate generated subnets against real-world requirements rather than producing subnets themselves. The moat comes from accumulated synthesis feedback across real capability descriptors, not from having interviewed the right experts. Decision: subnet library grown generatively from first principles; expert input used for validation and capability descriptor stress-testing, not for library seeding.

**Competitive landscape: partial replication risk is moderate, full replication is low**
Individual components of NeMo (Petri nets, knowledge graphs, unikernels, Claude API) are not novel in isolation and can be assembled into a superficially similar system by a well-funded team. The genuine moat is the integration: correctness-by-construction cannot be retrofitted onto an existing process tool — it must be the load-bearing structure from day one. The most credible near-term threat is a well-funded startup targeting the same regulated verticals with a more conventional tool (BPMN + formal verification + AI explanation) and better go-to-market. The most dangerous long-term threat is a company with existing enterprise relationships and graph infrastructure converging on the same problem space from a different architectural direction. The primary moat-building activity is accumulating synthesis feedback from real capability descriptors across target verticals. Decision: track competitive landscape; prioritise synthesis feedback accumulation and real deployment record over library size.

**Collaborative document editing via Claude Code in VS Code**
The prior workflow — Claude produces markdown files, user downloads and overwrites in the repository — is replaced by Claude Code running inside VS Code with direct read/write access to `~/Developer/nemo/`. Architecture documents are edited in place; both Claude Code and the user edit the same files. Install: `curl -fsSL https://claude.ai/install.sh | bash`, then install the Claude Code extension from the VS Code marketplace. Decision: Claude Code in VS Code as the canonical editing interface for architecture documents and code; this chat used for architectural reasoning and decisions; Claude Code used for file editing and implementation.

**Flutter web assets bundled into the unikernel binary**
Flutter web compiles to a static file bundle with no server-side runtime. Assets are embedded into the unikernel binary at compile time using `include_dir!`. The axum server serves both the Flutter bundle and `/api/*` routes from a single binary and a single network interface — consistent with the unikernel model of one binary, no filesystem dependency, no external process. TLS is terminated at the CDN edge (Cloudflare) in cloud deployments and at the unikernel via rustls in edge deployments; the same binary handles both, making TLS mode a runtime configuration. The WASM compilation target is preferred over JavaScript where browser support allows, and opens a path for `shared/core` components to run in-browser without network round-trips. Decision: single unikernel binary serves both Flutter UI and API; no separate web server at any deployment target; CDN edge for TLS and caching in cloud; unikernel-native TLS at edge.

**Constrained-ISA code generation for the NeMo unikernel core**
Because the NeMo unikernel runs a fixed, known workload — Petri net firing (bitwise operations on u128 bitfields), graph traversal (fixed-offset loads from fixed-width structs), and cryptographic operations (pre-validated opaque subnets) — the required instruction vocabulary is not ARM64 in general but a subset of approximately 20–25 instruction forms. This constraint makes the synthesis rules for code-generating nets exhaustively verifiable over their full vocabulary, which is not possible for a general-purpose ISA target. Memory safety (all offsets statically known and bounds-checked at synthesis time), register liveness (tracked as a reachability constraint, not solved as graph colouring), control flow integrity (branch targets are always validated subnet entry points), and stack discipline (fixed bounded frames, overflow inexpressible) are all enforced structurally. Cryptographic operations (HKDF, AES-256-GCM, Ed25519) remain as pre-validated opaque subnets — validated as units, not generated from primitives. The network stack (axum/Tokio) and slab allocator remain in Rust; these are infrastructure beneath the formal core. The bootstrapping Rust layer is measurably smaller than a general-purpose Rust application because the workload is fixed, making formal verification of the trusted base (Kani, Creusot) a tractable goal. Decision: constrained-ISA code generation targeting the ~25 instruction forms required by the NeMo formal core; cryptographic subnets as opaque pre-validated units; network stack and allocator remain in Rust; instruction-level synthesis rules exhaustively cover the vocabulary.

**All UI labels resolved by the Lexicon server — no app i18n layer**
Traditional app i18n (ARB files, `intl` package, locale detection in Flutter) is inconsistent with NeMo's core thesis that all knowledge is graph data. Labels are knowledge. Both domain labels (customer-owned) and NeMo platform labels (NeMo-owned) are stored as graph nodes and resolved by the Lexicon server unikernel via BCP-47 locale, with full CLDR plural support via the `icu_plurals` crate. The Flutter app has no locale detection logic and no ARB lookups in its permanent layer. Decision: Lexicon server is the sole localisation mechanism; `intl` package and ARB files are bootstrap exceptions only, pending Lexicon server delivery.

**ARB files and `intl` package are superseded as permanent architecture — now bootstrap exceptions**
The prior decision "all strings through `intl` ARB files from day one" (recorded in `client.md`) is reclassified as a bootstrap compromise, consistent with the supersession pattern. ARB files cover only the minimal set of strings required when the Lexicon server is unreachable (network error, connecting indicator, authentication screen labels, fatal crash message). This list is frozen and enumerated in `localisation.md`. Adding any string to ARB files requires a named Lexicon node target and an entry in `localisation.md`. Decision: prior ARB-first decision superseded; ARB files are a named bootstrap exception with a defined replacement path.

**`ResolvedLabel` carries `direction` and `script` per label**
A single UI response may contain labels in multiple languages with different text directions. Direction (LTR/RTL) and script (Latin, Arabic, Hebrew, etc.) are carried per label in `ResolvedLabel`, not inferred from a single session locale. The scaffold's layout direction is set by the user's locale (applied once at app root via `Directionality`); each label's text direction is applied locally by the rendering widget. These two concerns are independent and must not be conflated. Decision: `ResolvedLabel` carries `direction: TextDirection` and `script: Script`; scaffold direction and label direction resolved independently.

**Locale preference is a user graph node property**
User locale preference is a property on the user's graph node — consistent with the principle that user settings are graph data. No separate locale preference store. Locale precedence: user preference → org default locale → platform default ("en"). Resolved server-side on each label request. Decision: locale preference as a graph node property; precedence resolved in the Lexicon server.

**mistral.rs with Qwen3-14B-Q4K as the primary LocalUnikernelBackend**
Empirically validated by isolated head-to-head benchmark (March 2026) across four backends on Mac Mini M4 Pro 24GB, running identical prompt suites with quality scoring. Results:

| Backend | Quality | TTFT | Tok/s | JSON |
|---|---|---|---|---|
| mistral.rs Qwen3-14B Q4K | 100% | 382ms–1.9s | 28–900 | 3/3 |
| Claude Sonnet 4 (remote) | 93% | 1.5–2.2s | 37–78 | 3/3 |
| MLX Qwen3-14B-4bit | 83% | 135ms | 31–36 | 3/3 |
| Ollama qwen3:14b (unquantized) | 90% | 169ms | 2–3 | 3/3 |

Key findings: (1) Qwen3 thinking mode caused TTFT blowup on mistral.rs and `<think>` tag leakage on MLX in an earlier run — both fixed by disabling thinking mode at the API level (`"thinking": false`, `"enable_thinking": false`). (2) Ollama's low throughput is a quantization artifact — qwen3:14b auto-selected a high-precision variant; retesting with `qwen3:14b-instruct-q4_K_M` is expected to yield ~30 tok/s. (3) Claude's 93% quality score (vs 100% for mistral.rs) likely reflects the preamble heuristic firing on structured prompts, not a genuine quality gap — system prompt tightening is the fix. (4) MLX has the fastest TTFT (135ms) making it the correct OnDeviceBackend for latency-sensitive or offline scenarios. The benchmark tooling (`~/Developer/llm-bench/`) is a standalone Rust CLI producing JSONL results, maintained outside the monorepo. Decision: mistral.rs Q4K as `LocalUnikernelBackend`; MLX as `OnDeviceBackend`; Claude as `RemoteBackend`; fallback chain: local first → validate JSON → fall back to Claude on two consecutive validation failures.

**No custom LLM for NeMo**
Building a custom LLM from scratch for NeMo was explicitly evaluated and rejected. The cost (12–24 months, millions in compute) produces a model worse at the actual task — general-purpose semantic reasoning across arbitrary domains — than existing frontier models. A custom model would effectively reintroduce a hardcoded semantic layer, which the architecture deliberately avoids. The correct investment is in the interface between the formal core and the LLM: registry schema, graph query layer, prompt engineering, and context window management for large models. Fine-tuning on NeMo-specific output (subnet structural descriptions, registry hints) becomes viable once real synthesis data accumulates — on top of a capable base model, not instead of one. Decision: use frontier models (Claude as RemoteBackend, Qwen3 as local backends); revisit fine-tuning when synthesis feedback data is available.

**SemanticBackend abstraction with three-tier fallback**
The `SemanticActor` must not be coupled to a single LLM provider. A `SemanticBackend` trait with three concrete implementations — `OnDeviceBackend` (MLX, lowest latency, offline-capable), `LocalUnikernelBackend` (mistral.rs, highest local quality), `RemoteBackend` (Claude, quality fallback) — allows backend selection by configuration and enables graceful degradation. The fallback chain (local first → JSON validation → remote on failure) keeps the common case fast and private while ensuring correctness on complex inputs. Decision: `SemanticBackend` trait in the semantic layer crate; backend selection via runtime configuration; fallback chain with JSON validation as the gate.
